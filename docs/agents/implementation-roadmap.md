- **2026-07-20** — **six separate Slack apps, not one app + `chat:write.customize`.** Each agent role gets its own Slack app (own name, own avatar) so the team channel reads as six distinct teammates. Rejected the alternative of one shared app using `chat:write.customize` to override username/icon per message — too easy for agent code to forget to pass it, and identity would live in application code instead of Slack config. First one created: "Incus PO" for `Product_Owner`.
- **2026-07-20** — **two-way Slack via Socket Mode, not Slack's hosted MCP Server.** Slack's own MCP server (`mcp.slack.com`) was considered for letting agents read/post to Slack, but it authenticates as the human user who did OAuth, not as a distinct bot — would collapse all six personas into one voice. Built instead: `scripts/slack-bridge/bridge.py`, a per-role Socket Mode listener (no public webhook needed — fits the local-machine-only constraint) that answers `@mentions` with a headless `claude -p` call and posts the reply back in-thread. Wired for `Product_Owner` only so far; not yet validated against a real tool-heavy task. See `slack-bridge.md`.
- **2026-07-20** — **shared context, single responder.** Once a real second bot was imminent, added a passive `message` event handler to `bridge.py`: every bot now reads every channel message into an in-memory rolling buffer (`message.channels` event, now a required Slack app subscription, not optional) purely for context — it never triggers a reply. Only `app_mention` triggers a reply, and that reply's prompt now includes the recent buffer. Avoids both "bot is deaf outside direct mentions" and "all six bots answer every message" at once. See `slack-bridge.md §6`.
- **2026-07-20** — **portfolio survey done, four new functional roles proposed (not yet built).** Read across every repo in the portfolio; found three of six product lines (`nebulacast`, `roadsofttimes`, `stellar-attractor`) are primarily written content, not code, and none of the six existing roles owns drafting prose. Proposed `Content_Writer`, `Editor`, `Translator`, `Copywriter`, plus a `Domain_Consultant` pattern (first instance: `Astrophysicist` for Stellar Attractor). Sequencing suggested, none approved/built yet. See `proposed-new-roles.md`.

- **2026-07-20** — **persistent per-role Claude Code sessions in the Slack bridge.** `bridge.py` now keeps one continuous `claude` session per role (`--session-id` then `--resume`, id persisted in `scripts/slack-bridge/.sessions/<role>.txt`) instead of a stateless call per `@mention` — needed for the character personas to function as actual chat companions rather than resetting every message. Complements, doesn't replace, the shared-context buffer above. Flag: the `--session-id`/`--resume` flag behavior is assumed from Claude Code CLI docs, not verified live (no `claude` binary in the sandbox that wrote this) — check `claude --help` if resume doesn't behave as documented here.
- **2026-07-20** — **GitHub Projects board access was silently missing, now fixed.** PO tried answering a real Slack question ("which Stories did we just finish in Visual Composer") and hit two separate gaps in sequence: first a Claude Code permission wall (see the `.claude/settings.json` allow-list entries below), then — once that was fixed — discovered the remote GitHub MCP server's *default* toolset (used since nothing overrode it) doesn't include `projects` at all; none of the 44 available tools touched Project boards, only Issues/PRs/repos. Fixed at the source: `scripts/setup-github-mcp.sh` now registers the server with an explicit `X-MCP-Toolsets` header (`repos,issues,pull_requests,orgs,users,labels,projects`) instead of relying on the default. Adds three new tools: `projects_list`, `projects_get`, `projects_write` (GitHub's Jan 2026 consolidated Projects toolset). PAT scope was already sufficient (`setup.md` already specified org `Projects: Read and write`) — this was purely an MCP-server-side toolset selection gap, not a token permissions gap. Re-running the script requires a live `GITHUB_PAT` again since `.env` was deleted after first setup per the secrets-hygiene practice — either the original token (if saved somewhere outside the repo) or a freshly generated one with the same scopes.
- **2026-07-22** — **pilot succeeded.** `Product_Owner` → `Coder` → `Validator` ran end to end on a real Project 1 Story and shipped a real feature. Step 6 below is done.
- **2026-07-22** — **Slack bridge expanded past Product_Owner, then paused on a real blocker.** Mihal connected `Validator`, `Coder`, and `DevOps` too (four roles live now), and they've genuinely interacted with each other over Slack — the technology is proven. Currently paused: `Validator` found a real bug and `Coder` can't fix it live over Slack because of a ~300-second wait-for-response limit that appears to reset/misbehave in this flow. No plan to add more roles to Slack right now; next step here is fixing the timeout, not more rollout.
- **2026-07-22** — **`Publisher`'s external integration path decided: Claude in Chrome, not CDN/API.** Refines (doesn't reverse) the 2026-07-19 "external tooling deliberately deferred" entry above — `Media_keeper`'s CDN (R2) integration is still open/deferred as originally noted, but `Publisher` specifically will publish by driving the actual site/social UIs via Claude in Chrome browser automation (`mcp__claude-in-chrome__*`), not by provisioning CMS/social API credentials. One of the next tasks queued.
- **2026-07-22** — **branch/merge policy clarified, not changed.** Confirmed reading of the existing policy (`orchestration.md §4`, README §4): agents commit freely within their own working branch, no gate there — the PR + Mihal's human approval is a gate on *merging into `main`* specifically, not on in-branch work. The hook-enforcement-deferred decision above (2026-07-19) is about whether that merge gate is programmatically enforced (a hook blocking direct pushes to `main`) vs. relied on as a followed convention — still deferred, still just a convention for now.
- **2026-07-20** — **`run_all.py`: one process for every role, still one Slack app each.** Mihal didn't want to manage six open terminals. Refactored `bridge.py`'s per-role setup into a reusable `run_role(env_file)`, and added `run_all.py`, which discovers every `.env.<role>` at the repo root and starts each role's Socket Mode connection in its own thread within one process. Underlying architecture unchanged — still six separate Slack apps/tokens/sessions — this only collapses the terminal/process count. Also fixed a real thread-safety bug surfaced by this: the old code used `dotenv.load_dotenv()`, which writes into the shared process environment — fine for one role per process, but a race across threads (each role's tokens would stomp on the others'). Switched to `dotenv_values()` (returns an isolated dict per file, never touches `os.environ`); verified the isolation directly. `bridge.py` still works standalone, unchanged in behavior, for anyone who'd rather keep roles in separate processes.
